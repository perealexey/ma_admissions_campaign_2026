3. **Данные псевдонимизированы, но не публикуются в открытый доступ.** В отчётах нет ФИО/паспортов — только присвоенные университетом коды и баллы. ## 2. Архитектура: два независимых ручных контура

Не связывай сбор и публикацию в один автоматический конвейер. Два раздельных шага, оба запускаю руками:

- **Контур А (локально на Mac):** скачать отчёты → обработать → положить готовую нормализованную таблицу в папку репозитория.
- **Контур Б (публикация):** `git add / commit / push`. После пуша Streamlit Community Cloud, подключённый к репозиторию, сам перечитает данные.

Дашборд — отдельный код, пишется один раз и живёт в репозитории, просто читая актуальный файл данных.

## 4. Технические спецификации

### 4.1 Источник

- Поддомен **захардкодить как `priem47.hse.ru`** (вынеси в одну явную константу/переменную конфига). Номер поддомена меняется год к году — я поправлю его вручную одной строкой в следующую кампанию. Не пытайся определять его автоматически.
- Шаблон ссылки на отчёт: `https://priem47.hse.ru/ABITREPORTS/MAGREPORTS/FullTime/{ID}_{ТИП}.xlsx`, где `{ID}` — 11-значный код программы (начинается на `366`), `{ТИП}` ∈ {`Budget`, `Commercial`} (см. фильтр ниже).

### 4.2 Оглавление и его парсинг

Страница `magstats.html` строит таблицу программ через JavaScript — при простом скачивании данных в HTML нет (возвращается каркас с подписью «Разработка ООО Мельница Технологий»). Поэтому:

- **Я сам сохраняю оглавление через веб-инспектор** (уже отрисованный DOM) и кладу файл локально. Раз запуск ручной, это законный путь, не требующий headless-браузера. Буду давать тебе свежую копию при необходимости.
- **Особенность сохранённого файла:** инструмент Apple оборачивает HTML и экранирует его как текст (теги выглядят как `&lt;div&gt;`). Перед парсингом нужно снять экранирование (`html.unescape`).
- Внутри есть контейнеры кампусов вида `<div class="ms-campus" id="mill_local_AppCampus_N">`, каждый с заголовком `<h2 class="ms-campus__name">…</h2>`. Заголовки: «Москва», «НИУ ВШЭ - Санкт-Петербург», «НИУ ВШЭ - Нижний Новгород», «НИУ ВШЭ - Пермь».

**Фильтры при парсинге:**
- Брать только кампусы **«Москва»** и **«НИУ ВШЭ - Санкт-Петербург»**. Нижний Новгород и Пермь исключить.
- Брать только суффиксы **`Budget`** и **`Commercial`**. Суффикс **`ForeignCommercial` исключить** — иностранные коммерческие абитуриенты в анализ не входят (отдельная популяция).
- Для каждой программы вытащить привязку **ID → название программы** (и кампус). *Где именно в разметке лежит название относительно ссылки — нужно посмотреть на месте; это первая конкретная задача разведки. Я дам фрагмент HTML вокруг одной программы.*

**Ориентир по объёму** (из моей проверки сохранённого файла): всего 515 ссылок на отчёты по 4 кампусам; Москва — 372, СПб — 87, НН — 40, Пермь — 16. После фильтров (2 кампуса, 2 типа из 3) целевых файлов — порядка 459 минус иностранные коммерческие. Используй эти числа как грубую проверку, что фильтр отработал верно, а не как жёсткое ожидание.

### 4.3 Сбор файлов

- По собранному списку скачать xlsx последовательно, с короткими паузами.
- **Переживать отсутствие файла молча** (404). У программы может быть любое подмножество из {Budget, Commercial} — не у всех есть оба. Отсутствие файла — норма, не ошибка; логировать, но не падать.
- Сырые xlsx складывать в папку с датой прогона в имени (например, `raw/2026-07-15/`). Архив пока не используется для анализа (динамику по дням не строим), но сохраняется на случай, если позже понадобится — старые срезы не удалять.

### 4.4 Обработка

- **Сначала прочитай** приложенный `process_admissions.py` и пару реальных xlsx, прежде чем писать новый код. Переиспользуй логику нормализации, особенно подход с распознаванием колонок по именам/сигнатурам (в прошлом коде — `COL_SIGNATURES` с регэкспами), а не по позициям.
- **Шапка в этом году шире прошлогодней.** В реальных файлах есть дополнительные поля: целевые места, согласие на зачисление, статус участия в конкурсе, конкурс портфолио и др. Расширь сигнатуры под фактическую шапку. **Колонка «Конкурс портфолио» сейчас пуста** (конкурс ещё не прошёл) и наполнится позже — код не должен ломаться от пустых значений.
- **Выход — нормализованная «длинная» таблица**, НЕ широкая разреженная матрица прошлого года. Одна строка = один факт подачи: `(код_абитуриента, ID_программы, название, кампус, тип_места, приоритет, баллы, статус, портфолио, …)`. Это принципиальное решение: прошлогодняя «широкая» таблица (≈440 столбцов, почти все ячейки пусты) была медленной именно из-за формата. Длинный формат компактнее в разы, и пересечения программ из него считаются мерами.
- Сохрани результат в один файл, удобный для дашборда (например, Parquet), в папку репозитория.

### 4.5 Метрики

Считать из длинной таблицы:
- число заявлений по программам: бюджет / коммерц / всего;
- матрица пересечений абитуриентов между программами (кто подался в обе);
- распределение приоритетов по программам;
- **«индекс интересности»** программы — **зона обучения, см. ниже.** Требование к нему: НЕ простое среднее приоритетов. Среднее смещено (нишевую программу с немногими «первыми приоритетами» оно уравнивает с массовой «запасной»). Нужна поправка на число заявлений и разброс приоритетов. Конкретную формулу прорабатываю я с твоей помощью.

### 4.6 Визуализация и публикация

- **Streamlit.** Два разреза: (1) по программе — заявления, пересечения, индекс; (2) по абитуриенту — его программы, типы мест, приоритеты, результаты портфолио (когда появятся).
- Сначала локально (`streamlit run app.py`).
- Затем: приватный GitHub-репозиторий → Streamlit Community Cloud → доступ коллегам поимённо через их Google-аккаунты (приложение из приватного репозитория по умолчанию приватно). Обновление данных — обычным git-коммитом.

## 5. Режим работы и зоны обучения (ВАЖНО)

По умолчанию ты делаешь «тяжёлую» работу сам и **объясняешь, что делаешь и почему** — но не пишешь всё молча.

1. **git-цикл (`add` / `commit` / `push`, первичная настройка связи репозиторий ↔ Streamlit Cloud).** Мой уровень: немного работал с GitLab, поэтому общая идея (репозиторий, коммиты) мне не совсем чужая, но git как инструмент для меня неясен — командную строку git и специфику GitHub я уверенно не знаю, репозиторий на GitHub создаю и пушу впервые. Не объясняй с абсолютного нуля, но и не считай, что я владею командами: веди по реальному пробелу. В этих шагах объясни, что нужно сделать и зачем, дай мне написать команды самому, проверь и укажи на ошибки. Не выполняй git-команды за меня и не давай готовую последовательность для слепого копирования — веди меня, чтобы я понял логику.

2. **Расчёт «индекса интересности».** Это содержательное решение, которое я придумал, мне его и формализовать. Обсуди со мной варианты поправки на объём и разброс, дай мне написать расчёт, проверь корректность — но не подсовывай готовую формулу как данность.

3. **Развёртывание на Streamlit Cloud (регистрация и первый деплой).** Со Streamlit я не работал вовсе и регистрируюсь там впервые — и сам фреймворк, и Community Cloud для меня новые. На этапе хостинга не считай регистрацию, подключение репозитория и выдачу доступа коллегам рутиной: проведи меня по шагам с пояснениями, что и зачем я делаю в интерфейсе. (Написание кода самого дашборда на Streamlit при этом остаётся за тобой как heavy lifting — учебная часть здесь именно про развёртывание и устройство хостинга, а не про код виджетов.)

В этих зонах действуй как наставник, дающий разбор и подсказки, а не как генератор кода. (Так я раньше осваивал SQL — мне это подошло.) Остальное — парсинг HTML, нормализацию, код дашборда на Streamlit — бери на себя как heavy lifting, но с пояснениями по ходу.

Если в какой-то момент я скажу «просто доделай сам» — переключайся в обычный режим. Пока действует режим обучения на трёх зонах выше.
