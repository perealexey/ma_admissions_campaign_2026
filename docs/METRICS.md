# Метрики приёмной кампании (M1–M9) — справочник

Этот документ описывает слой метрик, реализованный в [`scripts/campaign_metrics.py`](../scripts/campaign_metrics.py)
(M1–M3, M5–M6, M9) и [`scripts/metrics.py`](../scripts/metrics.py) (M7–M8). Написан как
инструкция для возврата через год (следующая приёмная кампания) и как конспект
учебного разбора M4 — не только "что считаем", но и "почему так".

Живая копия последнего запуска метрик всегда лежит в `data/campaign_metrics_*.csv`
и пересчитывается заново на каждом срезе (динамика по дням не хранится).

---

## 0. Входные данные и что проверяется перед расчётом

- `admissions_long_YYYY-MM-DD.parquet` — длинная таблица заявлений, одна строка =
  одна подача `(student_id, program_id, program_name, campus, place_type, priority, …)`.
- `supply_demand_clean.csv` — `program_id, program_name, campus, n_budget_applicants,
  n_commercial_applicants, budget_places` (места приёма, сопоставленные со спросом
  по `program_id` — см. `scripts/match_supply_demand.py`).

Перед расчётом `validate_inputs()` проверяет:

1. `place_type` содержит только `budget`/`commercial` (без `ForeignCommercial`).
2. `campus` содержит только «Москва» и «НИУ ВШЭ - Санкт-Петербург».
3. Каждый `program_id` из длинной таблицы находится в `supply_demand_clean.csv`.

Отдельное важное допущение, не проверяемое кодом автоматически, а установленное
разведкой данных: **приоритеты в `budget` и `commercial` нумеруются НЕЗАВИСИМО**
(один и тот же человек может быть приоритетом №1 на бюджет и приоритетом №3 на
коммерцию той же программы). Поэтому почти все метрики считаются на срезе одного
`place_type`, а не на общей шкале приоритетов.

**Константа `K_SERIOUS = 2`** — порог «серьёзной» заявки (`priority <= K_SERIOUS`).
Захардкожена по опыту заказчика. **Перепроверить**, когда появятся баллы
портфолио/вступительных испытаний — возможно, порог стоит заменить на что-то,
завязанное на баллы, а не только на позицию в списке приоритетов.

## 1. Метрики — определения

Все метрики per-program. Там, где написано «budget/commercial раздельно» — это
**две колонки**, не сумма (см. допущение о независимой нумерации приоритетов выше).

### M1 — Заявления
Число строк (заявок, не людей) на программу, по `place_type`: `n_applications_budget`,
`n_applications_commercial`, `n_applications_total` (= сумма).

### M2 — Уникальные абитуриенты
`n_unique_applicants_total` — `nunique(student_id)` **по программе в целом**, budget и
commercial вместе. **Важно**: изначально это считалось раздельно по `place_type`
(`n_unique_applicants_budget/commercial`), но это оказалось тождественно M1 внутри
одного типа места — в данных нет ни одной повторной строки `(student_id, program_id,
place_type)`, поэтому «число заявок» и «число уникальных абитуриентов» внутри одного
`place_type` всегда совпадают. Разница появляется только на уровне программы целиком:
человек может подать и на бюджет, и на коммерцию одной программы — тогда в
`n_applications_total` он учтён дважды, а в `n_unique_applicants_total` один раз.
(Если в будущих срезах вдруг появятся дублирующие строки внутри одного `place_type` —
это будет заметно по расхождению M1 и M2, стоит проверять при каждом новом годе.)

### M3 — Серьёзный объём
Число заявок с `priority <= K_SERIOUS`, по `place_type` раздельно
(`n_serious_budget`, `n_serious_commercial`). Абсолютная величина, специально
размерозависима — она про объём серьёзного интереса, не про интенсивность
(интенсивность — это M6).

### M4 — Рейтинг интереса (только budget)
Доля заявителей, поставивших программу приоритетом №1, **со сжатием к общему
среднему** через эмпирический Байес. Полный разбор метода — в разделе 2 ниже,
это самая content-heavy метрика в наборе.

### M5 — Конкурс на место (только budget)
`competition_ratio_budget = n_applications_budget / budget_places`.
`budget_places == 0` → `NaN` (см. §3 про обработку краёв).

### M6 — Давление серьёзного спроса (только budget)
`demand_pressure_budget = n_serious_budget / budget_places` — сколько серьёзных
(`priority <= K_SERIOUS`) заявителей приходится на одно место. Значение <1 означает,
что серьёзных заявителей меньше, чем мест. **Сознательно не учитывает**
`enrollment_consent` (согласие на зачисление) — это чисто математическое отношение
объёма к предложению, не прогноз недобора.

### M7 — Распределение приоритетов
На программу × значение приоритета: число заявителей, budget/commercial раздельно.
Реализовано в `scripts/metrics.py::priority_distribution()`, файл
`data/metric_priority_distribution.csv`.

### M8 — Матрицы пересечений
Программа × программа, budget/commercial **раздельными матрицами**; ячейка = число
уникальных абитуриентов, подавшихся в обе программы этим типом места; диагональ =
число уникальных заявителей этого типа на программу (= M1 для этого `place_type`,
т.к. дублей внутри типа нет — см. M2 выше). Реализовано в
`scripts/metrics.py::program_intersections()`, файлы
`data/metric_program_intersections_budget.csv` / `_commercial.csv`.

Во вкладке дашборда «Сравнить программы» M8 дополнительно раскрывается
построчно: для выбранной пары программ показывается таблица общих
абитуриентов (их `student_id`/`reg_number` + приоритет по каждой из двух
программ). Приоритет на программу = минимум по всем спискам абитуриента
(бюджет и платное нумеруются независимо). Считается на лету из длинной
таблицы, отдельного файла-метрики нет.

### M9 — Согласованность / шумовая доля (диагностика, не для основного дашборда)
`noise_share_budget` — доля budget-заявок программы с `priority > K_SERIOUS`.
Отдельный диагностический показатель для разбора **по итогам** кампании (например,
эффект снятия лимита в 2 приоритета на бюджет). Файл помечен как
пост-кампания — не выводить в основной дашборд.

## 2. M4 подробно: почему сжатие и откуда берётся k

### Проблема, которую решает M4
Простая доля `cnt_p1/n` (доля заявителей, поставивших программу приоритетом №1) —
плохая оценка для программ с малым `n`. Пример: 2 из 3 первых приоритетов — это 67%,
но при n=3 это может быть чистый шум выборки, а не реальный «интерес» к программе.
Простое среднее приоритетов, которое обсуждалось изначально как альтернатива, страдает
той же болезнью и вдобавок уравнивает нишевую программу (мало заявок, но многие —
первым приоритетом) с массовой «запасной» (много заявок, в основном не первым
приоритетом).

### Идея решения: эмпирический Байес со сжатием (shrinkage)
Оценка каждой программы подтягивается к общему рыночному среднему `p0` тем сильнее,
чем меньше у неё данных (`n`). Сила подтягивания — параметр `k`: программа как будто
получает `k` дополнительных «виртуальных» заявок с усреднённой по рынку долей `p0`.
При `n → 0` интерес к программе → `p0` (неотличим от среднего рынка). При `n → ∞`
интерес → сырой `cnt_p1/n` (данных достаточно, чтобы им доверять, сжатие исчезает).

### Формулы и код (`scripts/campaign_metrics.py::compute_m4`)

```python
per_program = budget.groupby("program_id").agg(
    cnt_p1=("priority", lambda s: (s == 1).sum()),
    n=("priority", "size"),
)
```
Для каждой программы: сколько бюджетных заявителей поставили её приоритетом 1
(`cnt_p1`) и сколько всего бюджетных заявителей (`n`).

```python
p0 = per_program["cnt_p1"].sum() / per_program["n"].sum()
```
Глобальная доля первого приоритета — **сумма всех cnt_p1 / сумма всех n** по всему
рынку целиком, а не среднее долей по программам (иначе крошечные программы влияли бы
на `p0` непропорционально своему реальному весу).

```python
share_p1 = per_program["cnt_p1"] / per_program["n"]
big = per_program[per_program["n"] >= MIN_N_FOR_VARIANCE_ESTIMATION]  # n >= 10
share_p1_big = share_p1.loc[big.index]
```
Силу сжатия `k` оцениваем только на программах с `n ≥ 10` (`MIN_N_FOR_VARIANCE_ESTIMATION`) —
на малых `n` наблюдаемая доля слишком шумная, чтобы по ней вообще судить о том,
сколько в разбросе между программами «реального сигнала», а сколько шума.

```python
obs_var = share_p1_big.var(ddof=1)
```
Дисперсия наблюдаемых долей `share_p1` между программами — «сколько всего разброса
мы видим на рынке».

```python
samp_var = (share_p1_big * (1 - share_p1_big) / big["n"]).mean()
```
Теоретическая дисперсия из-за одной только конечности выборки (формула дисперсии
доли: `p(1-p)/n`), усреднённая по программам. Если бы у всех программ была одна и та
же истинная доля `p0`, весь наблюдаемый разброс `obs_var` объяснялся бы только этим
шумом.

```python
true_var = max(obs_var - samp_var, 1e-6)
```
**Метод моментов**: `истинный разброс = наблюдаемый разброс − ожидаемый шум выборки`.
Если `obs_var` даже меньше `samp_var` (весь разброс — шум, реального межпрограммного
сигнала нет), берём почти-ноль вместо отрицательного числа — и код печатает
предупреждение об этом случае.

```python
k = max(p0 * (1 - p0) / true_var - 1, 1.0)
```
Моделируем «истинные» доли программ как выборку из Beta-распределения со средним `p0`
и «псевдо-числом наблюдений» `k` (в терминах Beta(a,b): `a = k·p0`, `b = k·(1-p0)`,
`a+b = k`). У такого Beta-распределения теоретическая дисперсия равна
`p0(1-p0)/(k+1)`. Приравниваем её к оценке `true_var` и решаем относительно `k`.
**Интуиция**: маленький `true_var` (программы почти не отличаются друг от друга по
«настоящему» интересу) → большой `k` → сильное сжатие к среднему. Пол `≥1.0` —
защита от вырождения в отрицательное/нулевое `k`.

```python
per_program["desirability"] = (per_program["cnt_p1"] + k * p0) / (per_program["n"] + k)
```
Финальная формула — **апостериорное среднее Beta-Binomial модели**: как будто у
каждой программы было `k` дополнительных «наблюдений» с усреднённой долей `p0`, и
пересчитываем долю по объединённой выборке (`n` реальных + `k` виртуальных
наблюдений).

### Референсные значения (срез 2026-07-06)
- `p0 ≈ 0.3806`
- `k ≈ 9.74`
- Обучающая выборка для оценки `k`: 123 из 125 программ с бюджетными заявками
  (2 программы имели `n < 10` и не участвовали в оценке `true_var`, но сжатие к ним
  всё равно применяется).
- Эти значения **дрейфуют между срезами** — не хардкодить, брать из
  `data/campaign_metrics_meta.json` на каждый прогон.

### Проверка «здравого смысла» на срезе 2026-07-06
Программы-иллюстрации ортогональности M4 и объёма/конкуренции:
- «Медиапроизводство и медиааналитика» (СПб): небольшой объём (n=40), но высокая доля
  первого приоритета → ранг **6** из 125 в рейтинге интереса.
- «Медиаменеджмент» (Москва): самый большой объём заявлений (n=76) → лишь ранг **95**.
- Корреляция `desirability` (M4) с `competition_ratio_budget` (M5) на этом срезе:
  **≈ −0.15** — слабая отрицательная, подтверждает, что это разные, дополняющие друг
  друга оси, а не одно и то же число разными словами.

**Принципиально: M4 и M5/M6 не сводятся в один «магический» балл.** Держать
раздельными колонками — их слияние скрыло бы содержательные расхождения между
программами (интересная, но малоресурсная программа vs востребованная объёмом, но не
особо «первоприоритетная»).

## 3. Обработка краёв (обязательные правила)

- **53 программы без бюджетных мест** (`budget_places == 0`, срез 2026-07-06): M4, M5,
  M6 для них **не определены** — `NaN`, никогда не 0 («нет данных» ≠ «нулевой
  интерес»). В рейтинге интереса не показываются среди ранжированных — вынесены в
  отдельный файл `campaign_metrics_no_budget_places.csv`.
- **Деление на ноль в M5/M6** защищено — при `budget_places == 0` результат `NaN`, не
  исключение.
- **Вырождение `true_var` в M4** (вся вариация — выборочный шум): `k` уходит вверх,
  всё стягивается к `p0` — корректное поведение (нет межпрограммного сигнала), код
  печатает предупреждение при таком случае.
- **Программы-«близнецы»** (одно отображаемое имя, разные `program_id` — например
  «Аналитика данных для бизнеса и экономики» в СПб имеет два независимых конкурсных
  списка под одним именем): связка всегда идёт по `program_id`, никогда по имени —
  иначе это было бы ошибочное слияние двух разных конкурсов. См.
  `scripts/match_supply_demand.py` для полного разбора этого случая.

## 5. Как перезапустить на новом срезе

```bash
python3 scripts/campaign_metrics.py data/admissions_long_YYYY-MM-DD.parquet \
    data/supply_demand_clean.csv --snapshot-date YYYY-MM-DD
python3 scripts/metrics.py data/admissions_long_YYYY-MM-DD.parquet   # M7-M8
```

Оба скрипта берут дату из имени файла/аргумента — **не забыть передать актуальный
parquet/CSV**, иначе метрики посчитаются на устаревшем срезе (это уже случалось в
проекте, отмечено как техдолг).
