# Пайплайн приёмной кампании магистратуры ВШЭ — как это устроено

Документ исторической ценности: описывает **действующий** процесс сбора и анализа
данных приёмной кампании, каким он сложился к моменту первого написания (после
этапов 0–3 основного плана). Не хроника ошибок и доработок — только то, что
реально работает и как этим пользоваться в следующий раз (например, в следующую
кампанию). Дополняется по мере прохождения дальнейших этапов (git-цикл,
Streamlit-хостинг — см. отметку в конце файла).

Формулы и обоснование метрик — отдельно, в [`METRICS.md`](METRICS.md). Здесь —
только то, как данные превращаются из сырых xlsx в готовые таблицы.

---

## 0. Рамка проекта

- **Источник**: публичные xlsx-отчёты приёмной комиссии, поддомен `priem47.hse.ru`
  (номер меняется год к году — правится вручную одной константой).
- **Жёсткие ограничения**: заход на сайт не чаще раза в сутки, запуск только
  вручную («кнопкой»), никакого cron/launchd. Данные не публикуются в открытый
  доступ.
- **Архитектура**: два независимых ручных контура — (А) сбор+обработка локально,
  (Б) публикация в приватный репозиторий → Streamlit Cloud. Контур Б пока не
  пройден (см. отметку в конце файла).
- **Оглавление сайта** (`magstats.html`) строится через JavaScript — скачивается
  вручную через веб-инспектор (сохранённая копия DOM), не автоматическим браузером.

## 1. Этап 0 — разведка: список целевых файлов

**Скрипт**: [`scripts/parse_toc.py`](../scripts/parse_toc.py)

Вход — сохранённая (через веб-инспектор, Cocoa HTML Writer формат) копия
`magstats.html`. Обработка:

1. Вырезать `<body>…</body>`.
2. Снять Cocoa-обёртку (`<p>`/`<span>`, в которую Apple заворачивает исходный HTML
   как экранированный текст).
3. `html.unescape()` — получить реальную разметку страницы.
4. Разбить текст по контейнерам `ms-campus` (кампусы), внутри каждого — по строкам
   `ms-comp-table__row` (программы).
5. Из каждой строки взять название программы и ссылки вида
   `FullTime/{ID}_{Тип}.xlsx`, отфильтровать по кампусу (Москва, СПб) и типу
   (Budget, Commercial — ForeignCommercial исключается).

```bash
python3 scripts/parse_toc.py magstat/magstat_YYYY-MM-DD.html -o data/stage0_targets_YYYY-MM-DD.csv
```

Результат — CSV со списком целевых файлов: `campus, program_id, program_name,
place_type, filename, url`. На срезах 30.06/02.07/06.07 список стабилен: 178
программ, 302 целевых файла (Budget: 125, Commercial: 177).

**Важно про ID программы**: вопреки первоначальному предположению, ID программы
**не всегда** начинается на `366` (встречаются `367`, `369`, `370` и др.) —
фильтровать нужно строго по структуре ссылки, не по префиксу.

## 2. Этап 1 — сбор: скачивание xlsx

**Скрипт**: [`scripts/fetch_reports.py`](../scripts/fetch_reports.py)

Последовательно (не параллельно) скачивает файлы по списку из Этапа 0, с паузой
между запросами. Отсутствие файла (HTTP 404) — норма (не у всех программ есть оба
типа мест), логируется, не роняет скрипт. Сырьё складывается в
`raw/<дата прогона>/`, старые срезы не удаляются.

```bash
python3 scripts/fetch_reports.py data/stage0_targets_YYYY-MM-DD.csv
# опционально: --only data/some_ids.txt — скачать только подмножество program_id
```

Лог каждого прогона — `raw/<дата>/_fetch_log.csv`.

## 3. Этап 2 — нормализация: длинная таблица

**Скрипт**: [`scripts/normalize.py`](../scripts/normalize.py)

Читает все xlsx из папки `raw/<дата>/`, приводит к единой длинной таблице
(одна строка = одна подача заявления). Ключевые решения:

- Колонки сопоставляются по регэксп-сигнатурам (унаследовано от прошлогоднего
  `process_admissions_25.py`, адаптировано под расширенную шапку этого года), а
  не по фиксированной позиции.
- Строка-заголовок ищется по признаку `№ п/п` (не фиксированный номер строки);
  служебная строка-водяной знак сразу под заголовком (`~36913483045~` и т.п.)
  отсекается фильтром «первая колонка должна быть числом».
- Между колонкой приоритета и «Суммой конкурсных баллов» стоит переменное число
  (1–2) колонок вступительных испытаний с произвольным названием (не только
  «Конкурс портфолио» — бывает «Математика», «Собеседование», «Экзамен по …» и
  т.д.). Определяются позиционно, раскладываются в универсальные слоты
  `entry_test_1/2_name+score`.
- Идентификатор абитуриента — «Уникальный код поступающего» (`student_id`), не
  «Рег. номер»: проверено, что оба стабильны у одного человека между
  budget/commercial и разными программами, но первый — предназначенный для этого
  устойчивый ключ.
- Файлы с нулём заявок (программа есть, но никто не подал) — валидный пустой
  результат, не ошибка.
- `campus`/`program_name` берутся из списка Этапа 0 (join по `program_id`), не
  заново из ячейки A1 файла — чтобы результат был согласован с тем, что уже
  сверено глазами на стоп-гейте Этапа 0.

```bash
python3 scripts/normalize.py raw/YYYY-MM-DD data/stage0_targets_YYYY-MM-DD.csv \
    -o data/admissions_long_YYYY-MM-DD.parquet
```

Схема результата: `student_id, reg_number, program_id, program_name, campus,
place_type, priority, target_quota_priority, is_target_quota_applicant,
entry_test_1_name/score, entry_test_2_name/score, total_score, total_score_quota,
all_grades_positive, competition_status, enrollment_consent, docs_returned,
education_doc_type, has_contract, first_payment_made, orig_doc_submitted,
source_file`.

## 4. Сопоставление с местами приёма (КЦП)

**Скрипт**: [`scripts/match_supply_demand.py`](../scripts/match_supply_demand.py)

Отдельный источник — таблицы мест приёма (`moscow_supply.rtf`,
`stpeter_supply.rtf`), экспортированные из RTF. Обработка:

1. Конвертация `.rtf → .html` через `textutil` (сохраняет `<a href>` и корректно
   декодирует юникод — надёжнее ручного разбора RTF-эскейпов).
2. Строкой программы считается `<tr>`, где ячейка с названием содержит `<a>`
   (гиперссылку на страницу программы). Агрегатные строки (дисциплины,
   направления подготовки — суммы по нескольким программам) ссылки не имеют и
   отсекаются автоматически.
3. Сопоставление со спросом (`metric_applications_by_program.csv`) по
   нормализованному имени + кампусу, с курируемой картой псевдонимов
   (`SUPPLY_TO_DEMAND_ALIAS`) для опечаток в исходном RTF ВШЭ и известной
   аббревиатуры (ЦПМ).
4. Программы-«близнецы» (одно отображаемое имя, два разных `program_id`/конкурсных
   списка — подтверждено по `magstats.html`) сохраняются раздельно, не сливаются.
5. Три программы остаются без пары (объяснены в `KNOWN_SUPPLY_ONLY`: нет
   опубликованных отчётов или очно-заочная форма — вне scope).

```bash
python3 scripts/match_supply_demand.py other_files/moscow_supply.rtf \
    other_files/stpeter_supply.rtf data/metric_applications_by_program.csv
```

Результат: `supply_demand_clean.csv` (`program_id, program_name, campus,
n_budget_applicants, n_commercial_applicants, budget_places` — 178 строк, все
program_id получают значение) и `supply_demand_residual.csv` (объяснённые
исключения).

## 5. Этап 3 — метрики

**Скрипты**: [`scripts/metrics.py`](../scripts/metrics.py) (M7–M8, базовые срезы) и
[`scripts/campaign_metrics.py`](../scripts/campaign_metrics.py) (M1–M6, M9,
включая рейтинг желанности программ). Полное описание метрик, формул и разбор
метода сжатия — в [`METRICS.md`](METRICS.md).

```bash
python3 scripts/metrics.py data/admissions_long_YYYY-MM-DD.parquet
python3 scripts/campaign_metrics.py data/admissions_long_YYYY-MM-DD.parquet \
    data/supply_demand_clean.csv --snapshot-date YYYY-MM-DD
```

## 6. Сквозной порядок запуска (чек-лист на будущее)

1. Сохранить свежую копию `magstats.html` через веб-инспектор → положить в `magstat/`.
2. `parse_toc.py` → сверить список программ глазами (стоп-гейт).
3. `fetch_reports.py` → один раз в сутки, проверить `_fetch_log.csv`.
4. `normalize.py` → длинная таблица, проверить глазами (стоп-гейт).
5. (Опционально, если обновились места приёма) `match_supply_demand.py`.
6. `metrics.py` + `campaign_metrics.py` → метрики на свежем срезе.
